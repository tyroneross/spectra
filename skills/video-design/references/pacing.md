# Pacing & cut rate (quick card)

Full prose + research citations: `SOURCE-GUIDE.md` Part V, VIII, IX.

## Cut rate by channel
| Context | Cuts/min | Notes |
|---|---|---|
| TikTok/Reels/Shorts | 30–60 (1 per 1–2s) | 3s+ single cut often causes drop-off |
| LinkedIn feed | 15–30 (1 per 2–4s) | B2B tolerates more density per shot |
| YouTube product demo | 8–15 (1 per 4–7s) | Let the UI breathe |
| YouTube tutorial (long) | 6–12 (1 per 5–10s) | Cuts match topic transitions |
| App Store / Product Hunt | 15–25 (1 per 2–4s) | Hook-optimized |
| Social ad (paid) | 20–40 | First 3s needs 2+ cuts to stop the scroll |

## The 3-second social rule
Swipe/scroll decision happens within 3s. First cut lands by 1.5–2s; hook (visual or
audio) must land by 3s.

## Pattern interrupts
Every 20–40s in longer content, break the rhythm to recapture drifting attention:
visual (B-roll, zoom, talking-head↔screen swap), audio (music swell/cut, tone shift,
SFX), text (kinetic pop after plain VO), or reframing (new angle/speaker). **Time them
to content boundaries** — end of a point, never mid-sentence or mid-demo; a pattern
interrupt in the wrong place reads as choppy, not dynamic.

## One idea per beat
Each segment (between two cuts) carries exactly one idea. Splitting explain +
demonstrate + prove into one shot overloads working memory — the viewer disengages
rather than working harder (Cognitive Load Theory). Two ideas per frame are
acceptable only when tightly causally linked ("here's the input → here's the
output"); more than two, always split.

## Tight Edit vs. Breathe
- **Tight Edit** (MrBeast/YouTube growth): every second of dead air cut, relentless
  cuts — maximizes watch-time/completion. Wrong for enterprise/developer audiences,
  who read rapid cutting as hype.
- **Breathe** (Apple/Wistia): shots held longer, silence used deliberately — signals
  confidence. Fails on social, where the algorithm penalizes lower engagement velocity.
- **Apply:** Tight Edit for social (LinkedIn/Shorts/TikTok/Reels). Breathe for website
  hero, App Store previews, dev-docs demos. Blend for YouTube tutorials — breathe on
  UI demonstration, tight on commentary transitions.
- Both schools agree: no dead air where neither visual nor audio delivers new information.
